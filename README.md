# Final-project
An FPGA-Based Real-Time Interactive Switch, LED, and Seven-Segment Display Control System Using the PicoRV32 RISC-V Core and Memory-Mapped I/O

[cite_start]基於 PicoRV32 RISC-V 核心與 Memory-Mapped I/O 的 FPGA 即時互動式開關、LED 與七段顯示器控制系統 [cite: 66]。

5. 硬體設計說明
本專題之數位系統採用開源的 PicoRV32 (A Size-Optimized RISC-V CPU Core) 作為中央處理核心。在 SoC 架構的建構中，除採用核心外，其餘外設與整合皆為自行獨立設計與高度修改，核心貢獻包含：
(1) system_top.v (全系統頂層模組)
- 實作了正極性實體按鈕 (btnC) 至 CPU 核心所需低電位重設 (resetn) 的內部邏輯轉換。
- 建立 Memory-Mapped I/O (MMIO) 位址解碼器：將 CPU 發出的 32 位元實體位址進行分流，當存取開頭為 4'h8 (即 0x8000_XXXX) 時，精準導向週邊暫存器，包含 12'h000 (讀取指撥開關 sw)、12'h004 (寫入 led_reg 暫存器) 以及 12'h008 (寫入 seven_seg_reg 暫存器)。
(2) imem_dmem.v (內部整合式記憶體模組)
- 因應軟體編譯鏈啟動碼 (Startup Code) 所導致的位址跳轉行為，特別將內部 SRAM 陣列由原先的 4KB 深度大幅擴充至 64KB (16384 個 Word 空間)，以完美涵蓋 0x0000_8008 的指令定址區間。
(3) seven_seg_mux.v (七段顯示器驅動模組)
- 自行設計基於 18-bit 除頻器的動態掃描驅動電路，將 100MHz 降頻至約 400Hz 的最佳視覺流暢頻率。
- 引入「硬體防消隱/消殘影 (Anti-Ghosting) 技術」，在動態選通訊號 (an) 與掃描計數器切換時實施嚴格的同步閉鎖控制，徹底根除動態切換時常見的漏電與幽靈 8 殘影。

6. RISC-V 程式說明
由於本系統在硬體燒錄上電後，處理器核心會因編譯初始化環境而預設跳轉至 0x0000_8008 位址執行指令，因此本專題直接於硬體記憶體對應的陣列索引 8194 (0x8008 >> 2) 處埋伏核心程式機器碼，其對應的組合語言與功能如下：

ram[8194] = 32'h800007b7; // lui a5, 0x80000     (將 MMIO 基底位址 0x80000000 載入暫存器 a5)
ram[8195] = 32'h0007a603; // lw a2, 0(a5)        (自 0x80000000 位址讀取 4-bit 實體指撥開關狀態至 a2)
ram[8196] = 32'h00c7a223; // sw a2, 4(a5)        (執行 Store Word，將值寫入 0x80000004 控制實體 LED)
ram[8197] = 32'h00c7a423; // sw a2, 8(a5)        (執行 Store Word，將值寫入 0x80000008 控制七段顯示器)
ram[8198] = 32'hf11ff06f; // j 8194             (精準往回跳躍 16 Byte 偏移量，建立高速無限輪詢迴圈)

程式整體遵循標準的「初始化 ➡️ 讀取輸入 ➡️ 寫入週邊輸出 ➡️ 無限迴圈控制」流程，以高達數百萬次/秒的 Polling (輪詢) 機制，確保實體開關的物理變動能毫無延遲地反應在顯示硬體上。

7. 開發板與I/O配置
本專題所使用的硬體平台為 Digilent Basys 3 (晶片型號為 Xilinx Artix-7 XC7A35TCPG236-1)。
其實體週邊 I/O 資源配置與用途定義如下：
- sw[3:0] (實體指撥開關 SW0 ~ SW3)：提供使用者手動輸入 4 位元的二進位資料，用於設定欲顯示的十六進位數值 (0 ~ 15/F)。
- btnC (實體中央按鈕 T18 腳位)：定義為全系統的硬體異步/同步重設 (Reset) 控制端，按壓時系統暫存器清零，放開後立即重新啟動程式。
- led[3:0] (實體 LED 燈 LD0 ~ LD3)：用於即時對照目前開關被撥起與落下的物理狀態。
- an[3:0] (七段顯示器共陽極選通端)：輸出多工掃描致能訊號，以 400Hz 頻率輪流點亮四個顯示器。
- seg[6:0] (七段顯示器燈條 A ~ G)：輸出經字形解碼後的資料。在前三顆顯示器受控維持在「0」的防呆狀態下，最右側顯示器能精準、無殘影地呈現目前開關所代表的十六進位數字。

8. 實作與測試結果
系統成功通過實機測試，以下列出五個具代表性的測試案例 (Test Cases)，驗證軟硬體整合之正確性：

* 測試案 #1:
  - 輸入：sw[3:0] = 0000 (全關)
  - 預期輸出：LED全滅，七段顯示器顯示 "0000"
  - 實際結果：LED完全熄滅，七段顯示器精準顯示 "0000"。
  - 是否通過：是。
* 測試案 #2:
  - 輸入：sw[3:0] = 0001 (僅撥起 SW0)
  - 預期輸出：LD0 點亮，七段顯示器顯示 "0001"
  - 實際結果：LD0 即時亮起，七段顯示器個位數同步更新為 "1"，其餘為 0。
  - 是否通過：是。
* 測試案 #3:
  - 輸入：sw[3:0] = 0010 (僅撥起 SW1，數值為 2)
  - 預期輸出：LD1 點亮，七段顯示器顯示 "0002"
  - 實際結果：LD1 點亮，個位數精準轉變為 "2"；得益於防消隱技術，最高位數完全無「幽靈 8」殘影。
  - 是否通過：是。
* 測試案 #4:
  - 輸入：sw[3:0] = 0101 (撥起 SW0 與 SW2，數值為 5)
  - 預期輸出：LD0 與 LD2 點亮，七段顯示器顯示 "0005"
  - 實際結果：對應之兩顆 LED 同步亮起，七段顯示器流暢切換顯示 "0005"，字緣乾淨。
  - 是否通過：是。
* 測試案 #5:
  - 輸入：sw[3:0] = 1111 (開關全開，數值為 15/F)
  - 預期輸出：LD0 ~ LD3 全亮，七段顯示器顯示 "0015" (或依字形碼解碼為 000F)
  - 實際結果：4 顆 LED 達到最大亮度，七段顯示器亦同步穩定呈現正確的十六進位結尾值。
  - 是否通過：是。

9. 成果展示說明
(1) 啟動系統：將編譯完成的 system_top.bit 燒錄至 Basys 3 開發板後，板載 DONE 燈 (LD15) 亮起，CPU 隨即在不到 1 毫秒內自發完成啟動，並在位址 0x8008 處進入 Polling 監聽迴圈。
(2) 使用者輸入資料：使用者可隨時手動調整或撥動板上最右側的指撥開關 sw[3:0]。
(3) 系統顯示結果：上方實體 LED 燈號會即時呈現目前的開關二進位狀態，且最右側的七段顯示器會同步更新對應的十六進位數值。
(4) 正確性判斷標準：使用者可任意切換開關，若 LED 點亮位置與開關完全一致，且七段顯示器在變動時不需要按壓任何按鈕即可「即時即刻」更新數字，且不帶有任何先前數字遺留的模糊「幽靈 8」重疊暗影，即代表系統功能完全正確。

10. 問題、除錯與解決方式
本專題於實機開發與整合過程中，經歷了以下關鍵軟硬體問題的排除：
(1) 實體按鈕重設邏輯相反 (全系統死鎖)
- 找出問題：一上電燒錄後全板一片漆黑，唯有持續手動按住 btnC (T18 鍵) 時，開關與週邊才產生動作。原因為 Basys 3 實體按鈕未按時為低電位，而 PicoRV32 核心之 resetn 為低電位有效 (0 點重設)，導致處理器被永久凍結。
- 修正方式：於頂層電路 system_top.v 將實例化 CPU 的重設訊號改接為按鈕的取反訊號 (~btnC)，順利讓處理器正常起跑。
(2) CPU 啟動位址錯位 (Out of Bound)
- 找出問題：排除重設極性後系統依然卡死。經由將內部 mem_addr 的低 16 位元引腳強制對接到七段顯示器進行動態硬體診斷，發現 CPU 一上電就死鎖在 0x0000_8008 位址，此位址超出了原先規劃的 4KB 記憶體邊界。
- 修正方式：修改 imem_dmem.v，將 RAM 陣列深度由 4KB 大幅擴增至 64KB，並於對應的記憶體索引 8194 處，精密計算並手動寫入 -16 偏移量的 RISC-V 相對跳躍機器碼 (32'hf11ff06f)，迫使 CPU 在該位址建立穩定的即時 Polling 迴圈。
(3) Top Module 重名引發的 I/O Ports 塞爆錯誤 (316 腳位嚴重故障)
- 找出問題：在 Vivado 編譯 Place Design 階段跳出超載錯誤，指出設計包含了 316 個 I/O Ports，遠超晶片實體的 105 個引腳。檢查後發現，是在 system_top.v 內部的第一行代碼中，不小心將模組名誤寫成了 module seven_seg_mux，導致頂層宣告失效，Vivado 錯誤地將處理器內部上百條匯流排強行拉到實體引腳上。
- 修正方式：更正代碼第一行為 module system_top，並在 Sources 視窗按右鍵手動執行 Set as Top，重置 Run 快取後編譯順利通過。
(4) 動態掃描漏電殘影 (顯示器前面帶有模糊的幽靈 8)
- 找出問題：在切換到 2、3、4、5 等多數數字時，第一個七段顯示器會隱約疊加出一個未關乾淨的「8」字形暗影。此為典型動態多工掃描計數器切換速度與 an 訊號在高速下未能完美同步，導致前一週期之燈條餘電殘留（Ghosting）。
- 修正方式：重構底層驅動 seven_seg_mux.v，利用 18-bit 除頻器固定掃描頻率於穩定的 400Hz，並在多工選擇邏輯中嚴格加入 Common Anode 的同步消隱控制，成功消除物理殘影，使數值顯示清晰。
